# Design Patterns

> Reusable solutions to common software design problems. Each pattern provides a template for solving a particular type of challenge.

## Topics in this category
- [[Singleton Pattern]] — One instance to rule them all
- [[Factory Pattern]] — Object creation without specifying exact class
- [[Observer Pattern]] — Publish-subscribe for state changes
- [[Strategy Pattern]] — Interchangeable algorithms at runtime
- [[Decorator Pattern]] — Add behavior without modifying the original
- [[Adapter Pattern]] — Bridge incompatible interfaces
- [[Command Pattern]] — Encapsulate actions as objects
- [[Builder Pattern]] — Step-by-step complex object construction
- [[Prototype Pattern]] — Clone existing objects
- [[Proxy Pattern (Software)]] — Controlled access to another object
- [[Facade Pattern]] — Simplified interface to a complex subsystem
- [[Iterator Pattern]] — Sequential access without exposing internals
- [[State Pattern]] — Behavior changes based on internal state
- [[Template Method Pattern]] — Define algorithm skeleton, defer steps
- [[Mediator Pattern]] — Centralize complex communications
- [[MVC Architecture]] — Model-View-Controller separation
- [[Circuit Breaker Pattern]] — Prevent cascading failures
- [[CQRS Pattern]] — Separate read and write models
- [[Saga Pattern]] — Manage distributed transactions
- [[Pub/Sub Pattern]] — Decouple message producers and consumers
- [[Strangler Fig Pattern]] — Incrementally replace legacy systems
